# Getting Started

This chapter takes you from zero to a generated SystemVerilog module
in about five minutes. No design theory — just the commands and what
comes out.

## Install

`anvil` is a Rust crate. Clone, build, run:

<!-- book-test: skip — install instructions (placeholder git URL; not a generator example) -->
```bash
git clone <repo-url> anvil
cd anvil
cargo build --release
```

The binary lands at `target/release/anvil`.

**Every example in this book uses `cargo run --release -- …`** so it
works immediately after `git clone`, with zero extra setup — just
copy, paste, run. Cargo's default run target is `anvil`, so this
invokes the generator even though the repository also contains the
auxiliary `tool_matrix` binary (run that one with `cargo run --bin
tool_matrix -- …`).

> **Optional shorthand.** If you'd rather type `anvil …` than `cargo
> run --release -- …`, install it once and use the short form
> everywhere instead:
>
> <!-- book-test: skip — optional install shorthand; not a generator example -->
> ```bash
> cargo install --path .
> anvil --seed 42 --max-depth 2
> ```

## Your first module

A tiny starter, with everything small:

```bash
cargo run --release -- --seed 42 --max-depth 2 \
      --max-inputs 2 --max-outputs 1 \
      --flop-prob 0 --share-prob 0 \
      --construction-strategy sequential
```

Output:

```systemverilog
// Generated by anvil. Module: mod_42_0000
module mod_42_0000 (
    input  logic [30:0] i_0,
    input  logic [11:0] i_1,
    output logic [20:0] o_0
);

    wire [20:0] slice_0;
    wire [20:0] add_0;
    wire [20:0] mul_0;

    assign slice_0 = i_0[20:0];
    assign add_0 = slice_0 + slice_0;
    assign mul_0 = add_0 * add_0 * add_0;

    assign o_0 = mul_0;
endmodule
```

That's a valid, synthesizable SystemVerilog module. It will parse in
Verilator, elaborate in Yosys, and synthesize to a real netlist. It
does not compute anything *useful* — the logic is random — but it is
structurally legal and functionally non-trivial (the output genuinely
depends on `i_0`).

## Understanding the output

Walk through the example:

| Line                                  | What it is                          |
|---------------------------------------|-------------------------------------|
| `// Generated by anvil. Module: ...`  | Header comment; names seed + index. |
| `module mod_42_0000 (...);`           | Module declaration. Name is deterministic from `(seed, index)`. |
| `input logic [30:0] i_0`              | A 31-bit primary input. Width was randomly picked. |
| `output logic [20:0] o_0`             | A 21-bit primary output. |
| `wire [20:0] slice_0;`                | An internal signal. Named by kind: `slice_0` is the first `Slice` gate in the module. |
| `assign slice_0 = i_0[20:0];`         | `slice_0` is a bit-slice of `i_0`. |
| `assign add_0 = slice_0 + slice_0;`   | First `Add` gate. (Random logic — not meaningful.) |
| `assign mul_0 = add_0 * add_0 * add_0;` | First `Mul` gate, 3-arity (associative operator). |
| `assign o_0 = mul_0;`                 | Output driven by `mul_0`. |
| `endmodule`                           | End of module. |

The names follow a simple pattern: `<gate_kind>_<N>` where `<gate_kind>`
is the lowercase operator name (`and`, `or`, `xor`, `add`, `sub`, `mul`,
`eq`, `mux`, `slice`, `concat`, `shl`, `shr`, and a few more) and `<N>`
counts per kind from 0. Flops are `flop_<id>`. Reading `anvil` output
you can tell at a glance how much of each gate kind the module has.
See [Structural Rules — Rule 12](structural-rules.md) for the full
naming contract.

The names (`i_0`, `o_0`, `slice_0`, `add_0`, `mul_0`, …) are
**deterministic**: same seed plus same knobs always produces identical
text, byte for byte, on any platform.

## Reproducing output

```bash
cargo run --release -- --seed 42 --max-depth 2 --max-inputs 2 \
                      --max-outputs 1 --flop-prob 0 --share-prob 0
```

Run this twice. The output is byte-identical. This is a
non-negotiable `anvil` guarantee — it means any generated module can
be reproduced from its seed alone, forever, on any machine that can
build the tool.

## Generating many modules

For batches, use `--out`:

```bash
cargo run --release -- --seed 42 --count 100 --out ./generated/
```

This writes 100 files named `mod_42_0000.sv` through `mod_42_0099.sv`
into `generated/`, plus a `manifest.json` that records the seed, the
effective knobs, and a summary of each module (port counts, node
count, flop count).

## Dumping the effective configuration

`--dump-config` prints the effective knobs as JSON (after merging
defaults with CLI overrides), then exits:

```bash
cargo run --release -- --seed 42 --max-depth 4 --dump-config
```

Save it and replay later:

```bash
cargo run --release -- --dump-config > knobs.json
cargo run --release -- --config knobs.json --seed 42
```
